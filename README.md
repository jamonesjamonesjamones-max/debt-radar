# DebtRadar

**Code Health Auditor — Forensic technical debt analysis for Python, JavaScript, and TypeScript projects.**

> Traditional linters tell you when a semicolon is missing. DebtRadar tells you when your architecture is rotting.

DebtRadar is a forensic static analyzer that quantifies technical debt, detects over-engineering, and alerts on patterns of accidental complexity — all locally on your machine.

---

## What is DebtRadar?

DebtRadar builds an Abstract Syntax Tree (AST) of every file and measures the architectural health of your repository. It is not a linter — it is a code radiologist.

### What it detects

1. **Monolithic files** (>800 lines) — files that should be split into modules
2. **Extreme cyclomatic complexity** (>7 nesting levels) — hard-to-follow code
3. **Over-engineering** (functions with >20 branches) — unnecessarily complex code
4. **Magic numbers** — numeric literals without descriptive names
5. **Accumulated TODOs** — documented but unresolved technical debt
6. **Historical evolution** — how code quality changes between commits
7. **Author attribution** — who introduced each violation (via git blame)

### What it does NOT detect

DebtRadar analyzes structural patterns, not semantics:

1. **AI-generated code** — We detect symptoms of complexity, not their origin. Complex code can come from AI, tired humans, tight deadlines, or lack of code review. We cannot distinguish between them.
2. **Runtime bugs** — A 50-line file with a perfect A score can still have a critical bug. Use linters, type checkers, and tests for runtime issues.
3. **Security vulnerabilities** — We do not search for injections, CVEs, or dependency flaws. Use specialized tools like Snyk, Bandit, or Semgrep.

### Why "Anti-Bloat" and not "Anti-Vibe-Coding"?

The term "vibe-coding" is associated with AI-generated code, but we cannot prove that code was generated by AI. What we can detect is the **symptom**: unnecessarily complex, monolithic, or bloated code.

The real problem is **accidental complexity**, regardless of its origin. A human under pressure can write code just as complex as an unsupervised AI. Our approach is to detect the symptom and let the team decide the cause.

## Features

| Feature | Description |
|---------|-------------|
| **Heat Map** | Treemap of every file: size = lines of code, color = health grade (A-F) |
| **Debt Radar** | 4-axis radar of complexity, TODOs, magic numbers, and oversized files |
| **Hall of Shame** | Paginated ranking of worst files with exact violation locations |
| **Code Viewer** | Modal inspection with severity-coded violations and code context |
| **Action Plan** | Files ranked by impact per minute of effort — best ROI first |
| **Dependency Graph** | Interactive force-directed graph of import relationships, color-coded by health |
| **Scan Comparison** | Side-by-side diff against previous scans — what improved or regressed |
| **Portfolio Dashboard** | Multi-project overview for teams managing many repositories |
| **Focus Mode** | One-click toggle to hide visualizations and show only action items |
| **Scan History** | SQLite-based tracking of score evolution across multiple scans |
| **Git Blame** | Who introduced each violation, per file and per author |
| **Export** | Self-contained HTML report and dynamic SVG badge for READMEs |
| **File Upload** | Drag-and-drop single-file analysis without scanning a full project |
| **CLI Mode** | Terminal scans with `--output json` for CI/CD pipelines |

## Tech Stack

| Layer | Technology |
|-------|-----------|
| **Backend** | Python 3.11+, FastAPI, Uvicorn |
| **Parser Engine** | Tree-sitter (prebuilt packages for Python, JS, TS, TSX) |
| **Concurrency** | ProcessPoolExecutor with per-file 30s timeout |
| **Filtering** | pathspec for `.gitignore` support |
| **Frontend** | React 18, Vite, TailwindCSS |
| **Visualization** | Recharts (Treemap, Radar, Line, Bar) |
| **Real-time** | Server-Sent Events (SSE) for live scan progress |
| **History** | SQLite for cross-scan comparisons and trend tracking |

## Quick Start

### Requirements

- Python 3.11+
- Node.js 18+
- npm or yarn

### 1. Clone

```bash
git clone https://github.com/jamonesjamonesjamones-max/debt-radar.git
cd debt-radar
```

### 2. Backend

```bash
cd backend
python -m venv venv
source venv/bin/activate        # Linux/Mac
# .\venv\Scripts\activate       # Windows
pip install -r requirements.txt
uvicorn main:app --reload --host 127.0.0.1 --port 8000
```

### 3. Frontend (new terminal)

```bash
cd frontend
npm install
npm run dev
```

### 4. Open the app

Navigate to **http://127.0.0.1:5173**

1. Paste the absolute path to your project
2. Adjust parallelism (1-8 workers)
3. Click **"Scan"**
4. Watch real-time progress, then explore the interactive dashboard

### Try sample data instantly

Click **"Explore Sample Report"** on the welcome screen to see a live demo with pre-loaded data — no scan needed.

## Scoring System

Each file starts at **100 points**. Deductions are applied:

| Factor | Condition | Deduction |
|--------|-----------|-----------|
| God files | >1000 lines | -40 |
| | >500 lines | -20 |
| Nesting | >6 levels | -30 |
| | >4 levels | -15 |
| TODOs | Per 100 lines | up to -20 |
| Magic Numbers | Per 100 lines | up to -25 |
| Complex functions | >15 branches | -25 |
| | >10 branches | -10 |

The repository score is the **line-weighted average** of all file scores.

### Grades

| Grade | Score | Meaning |
|-------|-------|---------|
| **A** | 90-100 | Clean Code |
| **B** | 75-89 | Minor Debt |
| **C** | 60-74 | High Debt |
| **D** | 40-59 | Fragile Code |
| **F** | 0-39 | Biohazard |

## Architecture

```
┌──────────────────────────────────────────────────────┐
│           FRONTEND (React + Vite)                    │
│   Path Input  •  Dashboard  •  Charts  •  Slider     │
│   EventSource (SSE) for real-time progress           │
└──────────────────────┬───────────────────────────────┘
                       │ HTTP POST + SSE Stream + CORS
┌──────────────────────▼───────────────────────────────┐
│           BACKEND (FastAPI + Uvicorn)                 │
│   • POST /api/scan       → creates background job     │
│   • GET /scan/{id}/stream → SSE progress events       │
│   • .gitignore filtering (via pathspec)               │
│   • Sensitive path protection (cross-platform)        │
│   • Rate limiting (max 5 concurrent scans)            │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────┐
│        ANALYSIS ENGINE (ProcessPoolExecutor)          │
│   • safe_process_file() with 30s timeout              │
│   • Tree-sitter AST parsing (PY/JS/TS/TSX)           │
│   • Cyclomatic complexity + nesting detection         │
│   • Magic number and TODO/FIXME pattern matching      │
│   • Normalized scoring + file-level aggregation       │
└──────────────────────────────────────────────────────┘
```

## Known Limitations

- **Languages**: Python, JavaScript, TypeScript, TSX only
- **Single root `.gitignore`**: Does not support nested `.gitignore` files
- **File limit**: Maximum 5,000 files per scan
- **Size limit**: Files > 1MB are skipped
- **Static analysis**: Does not detect runtime issues, only structural patterns
- **Scoring calibration**: Thresholds may need tuning per project ecosystem
